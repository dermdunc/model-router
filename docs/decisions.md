# Decisions: Model Router

## ADR Log

| Date | Decision | Rationale |
|---|---|---|
| 2026-07-22 | Initial scaffold as factory-output | A real CLI for the cost-aware model-routing decision: given a task's data sensitivity and complexity, recommend a model tier and the controls that come with it. |
| 2026-07-22 | Extracted the routing table from `agentic-tekton/src/components/RoutingWidget.astro` into `lib/route.mjs`, and fixed a real bug found while doing it: sensitivity now actually caps the recommended tier | The widget's own code comment claimed "high sensitivity can pull it [the tier] back toward local," but the shipped code (`let t = state.complexity`) never applied sensitivity to the tier at all. `lib/route.mjs`'s `SENSITIVITY_TIER_CAP` implements the documented intent for real: `tierIdx = min(complexityIdx, cap[sensitivityIdx])`. Same fix ported back to the live widget on `agentic-tekton` â€” see that repo's own decisions.md. |
| 2026-07-22 | Doubt-driven-development pass (single-model, `Explore` agent) found and fixed a second, real bug: the controls shown didn't always match the tier recommended | For `(sensitivity=low, complexity=medium)`, the tool recommended the mid-tier but printed only the local tier's controls â€” missing the eval gate, cost cap, and mandatory human review the recommended tier itself calls for. Root cause: the controls-level threshold used a separately-computed `complexity >= 2` check that didn't match the threshold `tierIdx` itself uses to reach level 1. Fixed by driving the controls threshold off the actual `tierIdx`, not a parallel complexity check. The same review also found the original "exhaustive" grid test was tautological â€” its only assertion (`tierIndex <= complexityIndex`) holds true by construction of `Math.min()` regardless of whether the controls logic is broken, verified by simulating the pre-fix bug and confirming the old test still passed all 9 cases. Replaced with a full 9-case table asserting both tier and controls per combination. |
| 2026-07-22 | Cross-model review (`codex exec --sandbox read-only`, gpt-5.5) found and fixed 4 more real issues | `CONTROLS`/`TIERS` were mutable shared module state, and `route()` returned the live array reference, so a caller mutating `result.controls` (or the exported `CONTROLS` directly) would silently corrupt every later call in the same process â€” fixed by freezing both constants (`Object.freeze`, each entry individually since freeze is shallow) and returning a copy from `route()`. The new 9-case test table used `CONTROLS`/`TIERS` as its own oracle, so a content regression (e.g. "Cost cap" quietly dropped) would still pass every case â€” added two independent, hand-written content assertions that don't import from `route.mjs`'s own exports for their expected values. The newly-added `--flag=value` CLI syntax wasn't exercised by any test or CI step â€” added to the CI smoke test. `package.json` had no `files` allowlist, so a future `npm publish` would ship this repo's internal Hekton scaffolding (`.hekton/`, `mind-palace/`, docs) into the package â€” added a `files` allowlist and a `LICENSE` (MIT, matching the already-declared license field). |
